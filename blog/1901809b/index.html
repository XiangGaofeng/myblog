前后端分离为什么不能用session

在前后端分离的架构下，前端与后端不再共享session，因为后端应用是无状态的，每次请求都是新的请求，不保存上次请求的状态，所以前端每发送一次请求，都需要从新获取用户信息进行验证。

前后端完全分离后，由于前端不再接受后端传来的渲染好的HTML页面，而是后端只提供RESTful API接口，前端通过AJAX请求后台返回的JSON数据自己进行渲染，所以后台不再需要用session来保存状态，前端使用session效果不好甚至有风险：

  • session实现复杂，需要后台程序开发共享session机制，前端需要保持session的一致性。
  • session空间大小有限制，过多的数据会占用大量的内存，并且session可以被其他应用篡改。

而采用令牌(token)作为身份标识，则前后端都不再需要保存状态。后台验证通过后，生成一个令牌，将其返回给前端。后端不再保存令牌信息，前端保存令牌并在随后的请求中携带令牌，并随请求传递给后端。后端收到请求后验证携带的令牌，验证通过后则认为是合法请求，并返回所需要的结果。

过滤器Filter和拦截器Interceptor区别

Filter和Interceptor都是Java Web开发中常用的拦截器，它们的作用都是对请求进行拦截和处理，但是它们之间还是有一些区别的。

  1. Filter是Servlet规范中的一种过滤器，是在Servlet容器中实现的，而Interceptor是Spring MVC框架中的一种拦截器，是在Spring容器中实现的。

  2. Filter是在请求到达Servlet之前或之后进行拦截和处理，可以对请求和响应进行修改，但是不能访问Controller中的方法和数据；而Interceptor是在请求到达Controller之前或之后进行拦截和处理，可以访问Controller中的方法和数据，可以对请求和响应进行修改，也可以在Controller方法执行之后对返回结果进行修改。

  3. Filter可以配置在web.xml中，也可以通过注解方式进行配置，而Interceptor只能通过注解方式进行配置。

  4. Filter是Java Web开发中的标准组件，可以用于任何Java Web应用程序中，而Interceptor是Spring MVC框架中的组件，只能用于Spring MVC应用程序中。

总的来说，Filter更加通用，可以用于任何Java Web应用程序中，而Interceptor则更加灵活，可以访问Controller中的方法和数据，对返回结果进行修改。在实际开发中，可以根据具体的需求选择使用Filter或Interceptor来实现拦截和处理。

Java的函数是值传递还是引用传递

Java 中的函数参数传递是值传递。这意味着当你将一个变量作为参数传递给一个函数时，函数会得到这个变量的一个副本，而不是这个变量本身。因此，如果在函数中修改了参数的值，那么原始变量的值不会受到影响。但是，如果参数是一个对象，那么副本将包含对该对象的引用，因此在函数中修改对象的属性将影响原始对象的属性。也就是说，虽然 Java 的函数参数传递是值传递，但是对于对象类型的参数，其实是传递了一个引用的副本。

CMD和ENTRYPOINT的区别

CMD 和 ENTRYPOINT 是 Dockerfile 中两个常用的指令，它们都可以用来定义容器启动时要执行的命令，但是它们有一些不同之处。

为什么用docker而不是虚拟机

使用 Docker 相对于虚拟机的优点主要有以下几个方面：

  1. 更轻量：Docker 容器与宿主机共享操作系统内核，因此 Docker 容器相对于虚拟机来说更加轻量级，启动和停止更快。

  2. 更快速：Docker 容器的启动时间可以在秒级别内完成，而虚拟机需要几分钟才能启动。

  3. 更灵活：Docker 容器可以在任何环境中运行，而虚拟机则需要在特定的虚拟化平台上运行。

  4. 更易于管理：Docker 容器的管理更加简单，可以通过 Docker 命令行工具或者 Docker Compose 来管理容器。

因此，相对于虚拟机，Docker 更加轻量级、快速、灵活和易于管理，因此在很多场景下更为适用。
